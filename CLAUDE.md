# ЦСУПД — Цифровая система учёта и планирования деталей

## Контекст проекта

Заказчик: **УЗГА** (Уральский завод гражданской авиации).  
Разработчик: команда УрФУ УПИШ, направление «Машиностроение», 2 курс.  
Объект автоматизации: **участок комплектования деталей** перед сборкой авиадвигателей **ТВ2-117** и **ТВ3-117**.

### Проблема (зачем это вообще нужно)

Сейчас на участке нет цифрового учёта. Детали списываются вручную на бумаге или не списываются вообще. Из-за этого:
- никто не знает реальный остаток деталей в реальном времени
- детали пропадают без фиксации (неучтённое изъятие)
- дефицит обнаруживается только в момент сборки → простой → срыв плана (20 двигателей/месяц)

### Цель MVP

Веб-приложение для учёта прихода и расхода деталей со светофорной индикацией остатков. Должно работать в браузере на планшете и ПК, интерфейс на **русском языке**.

## Модули системы (что нужно реализовать)

### 1. Дашборд остатков (главная страница)
- Таблица всех деталей с колонками: Артикул, Наименование, Двигатель, Ячейка хранения, Остаток, Статус
- **Светофор** по каждой позиции:
  - 🟢 Зелёный — остаток выше жёлтого порога (норма)
  - 🟡 Жёлтый — остаток между жёлтым и красным порогом (уведомление кладовщику)
  - 🔴 Красный — остаток ниже красного порога (критично, уведомление мастеру)
- Красные и жёлтые позиции — **вверху списка**
- Поиск и фильтр по наименованию, артикулу, типу двигателя
- Автообновление каждые 5 минут

**Формулы порогов:**
```
Плановый дневной расход = Плановый месячный расход / 22
Красный порог = Дневной расход × 1 день (срок поставки)
Жёлтый порог = Дневной расход × (1 + 3) дня (срок + страховой запас)
```

### 2. Модуль прихода деталей (только кладовщик и выше)
- Форма: выбор номенклатурной позиции, количество, ячейка хранения, исполнитель
- После сохранения — остаток обновляется немедленно
- Запись в историю операций

### 3. Модуль расхода деталей (все роли)
- Сборщик выбирает: номер двигателя → этап сборки → деталь → количество
- Система не даёт списать больше чем есть (с предупреждением)
- Максимум 3 нажатия от главного экрана до подтверждения
- Запись в историю операций

### 4. Заявки на пополнение (кладовщик и выше)
- Создать заявку из карточки позиции (кнопка «Заказать»)
- Статусы заявки: Создана / В работе / Выполнена
- Список всех заявок с фильтром по статусу

### 5. История операций
- Все операции: дата/время, тип (приход/расход), позиция, количество, исполнитель, двигатель
- Фильтрация по дате, типу операции, позиции, двигателю
- Экспорт в Excel (.xlsx)

### 6. Справочники (только администратор)
- Номенклатура деталей (артикул, наименование, тип двигателя, ед. измерения, плановый месячный расход)
- Адресное хранение (код ячейки в формате: Стеллаж-Полка-Ячейка, например `С1-П2-Я3`)
- Пользователи и роли
- Пороговые значения светофора по каждой позиции

## Структура базы данных

```sql
-- Номенклатура деталей
CREATE TABLE parts (
    id INTEGER PRIMARY KEY,
    article TEXT NOT NULL,          -- артикул (обозначение)
    name TEXT NOT NULL,             -- наименование
    engine_type TEXT,               -- 'ТВ2-117' или 'ТВ3-117' или 'оба'
    unit TEXT DEFAULT 'шт',         -- единица измерения
    monthly_plan INTEGER DEFAULT 0, -- плановый расход в месяц (на 20 двигателей)
    yellow_threshold INTEGER,       -- жёлтый порог (если NULL — рассчитывается автоматически)
    red_threshold INTEGER            -- красный порог (если NULL — рассчитывается автоматически)
);

-- Ячейки хранения
CREATE TABLE storage_cells (
    id INTEGER PRIMARY KEY,
    cell_code TEXT UNIQUE NOT NULL, -- код ячейки, например С1-П2-Я3
    shelf_number TEXT,
    level_number TEXT,
    cell_number TEXT,
    notes TEXT
);

-- Остатки (привязка детали к ячейке)
CREATE TABLE stock (
    id INTEGER PRIMARY KEY,
    part_id INTEGER REFERENCES parts(id),
    cell_id INTEGER REFERENCES storage_cells(id),
    quantity INTEGER DEFAULT 0,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- История движений
CREATE TABLE movements (
    id INTEGER PRIMARY KEY,
    part_id INTEGER REFERENCES parts(id),
    cell_id INTEGER REFERENCES storage_cells(id),
    operation_type TEXT,            -- 'приход' или 'расход'
    quantity INTEGER,
    engine_number TEXT,             -- номер двигателя (для расхода)
    assembly_stage TEXT,            -- этап сборки (для расхода)
    operator TEXT,                  -- ФИО / логин исполнителя
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    notes TEXT
);

-- Заявки на пополнение
CREATE TABLE replenishment_requests (
    id INTEGER PRIMARY KEY,
    part_id INTEGER REFERENCES parts(id),
    quantity INTEGER,
    status TEXT DEFAULT 'Создана',  -- Создана / В работе / Выполнена
    created_by TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    completed_at DATETIME
);

-- Пользователи
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    username TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    full_name TEXT,
    role TEXT DEFAULT 'sborshik'    -- sborshik / kladovshik / master / admin
);
```

- Язык: только **русский**

## Визуальный стиль (обязательно соблюдать)

- **Тёмная тема** — тёмный фон (#1a1a2e или аналог), светлый текст
- **Боковое меню** слева с разделами: Главная, Остатки, Приход, Расход, Ячейки, Детали, Заявки, Журнал операций, Отчёты, Настройки
- **Три карточки вверху**: Критический остаток (красный), Внимание (жёлтый), Норма (зелёный) — с количеством позиций
- **Список остатков**: артикул, наименование, количество, порог — с цветной точкой светофора слева
- **Быстрые действия**: Приход, Расход, Поиск детали (QR-сканер в MVP не реализовывать)
- **Последние операции** справа внизу — лента с +/- и цветом
- Кнопки минимум 44×44px (работа с планшетом)
